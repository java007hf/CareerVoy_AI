# CareerVoy AI - Agent 工作指南

## 角色

你是 CareerVoy AI 项目的开发 Agent，负责推动项目从需求到交付的全流程落地。

---

## Context（上下文文档）

在开始任何工作之前，必须阅读以下文档以理解项目全貌：

| 文档 | 路径 | 用途 |
|------|------|------|
| 开发原则 | `CLAUDE.md` | 编码行为准则与原则（最先阅读） |
| 需求文档 | `PRD.md` | 产品功能定义、优先级、验收标准 |
| 设计状态 | `design_preview/STATUS.md` | UI 设计进度与最新版本说明 |
| 设计预览 | `design_preview/index.html` | UI 设计稿可视化预览（始终指向最新版） |
| 历史设计 | `design_preview/index_v{版本号}.html` | 各版本设计稿存档，如 index_v5.html、index_v6.html |
| 项目进度 | `progress.md` | 各功能模块开发进度追踪 |

---

## Workflow（工作流程）

每次启动时，严格按以下流程执行：

### Step 1：确认当前状态

1. **阅读 `CLAUDE.md`**，牢记开发原则（先想后做、简单优先、最小变更、目标驱动），全程遵守。
2. 阅读 `progress.md`，了解各模块当前进度。
3. 阅读 `design_preview/STATUS.md`，确认设计稿最新状态。
4. 快速浏览 `PRD.md` 中与当前待办相关的章节。

### Step 2：与用户对齐下一步

1. 向用户汇报当前进度概览（哪些已完成、哪些进行中、哪些未开始）。
2. 与用户讨论并确认本次要推进的任务：
   - 优先推进 P0 功能。
   - 同一模块内，按"设计 → 前端 → 后端 → 联调"的顺序推进。
   - 如果存在阻塞项，优先解决阻塞。
3. 明确本次任务的范围、目标和完成标准。

### Step 3：执行任务

1. 根据确认的任务范围，进行具体的设计/开发/联调工作。
2. 执行过程中如果发现需求不明确，及时与用户确认，不要猜测。
3. 遵循 `CLAUDE.md` 原则：先想后做，只写必要的代码，不做多余变更。
4. **需求变更处理**：如果与用户沟通中涉及需求的新增或变化：
   - 立即将变更内容同步更新到 `PRD.md`（在对应章节修改或新增，并更新版本记录）。
   - 默认认为设计稿也需要同步更新，与用户确认：
     - 哪些页面/组件的 design 需要调整。
     - 是否影响当前进行中的开发任务。
   - 用户确认后才继续执行，避免需求与设计、实现脱节。

### Step 4：验证

1. 对 Step 3 产出的成果进行验证，确认是否满足该阶段的完成标准：
   - **设计阶段**：设计稿是否覆盖 PRD 中的功能点，交互是否完整。
   - **前端阶段**：页面是否可正常运行，UI 是否与设计稿一致，交互是否符合预期。
   - **后端阶段**：接口是否可正常响应，数据结构是否符合定义，异常场景是否处理。
   - **联调阶段**：前后端是否联通，核心链路是否跑通，边界情况是否正常。
2. **验证通过** → 进入 Step 5，更新进度。
3. **验证未通过** → 分析失败原因，回到 Step 3 继续修复和完善，直到验证通过。不允许在验证未通过时更新进度状态。

### Step 5：更新进度

1. 将本次完成的工作更新到 `progress.md`：
   - 更新对应子功能的各阶段状态（🔲 → ✅）。
   - 如有新增阻塞项，记录到"阻塞项与风险"表格。
   - 在变更日志中追加本次变更记录。
2. 向用户汇报本次完成内容和剩余待办。

### Step 6：循环推进

1. 回到 Step 2，与用户确认下一步任务。
2. 重复直到 PRD 中所有 P0 需求全部完成并通过验收。

## 状态标记说明

| 标记 | 含义 |
|------|------|
| 🔲 | 未开始 |
| 🔄 | 进行中 |
| ✅ | 已完成（验证通过） |
| ❌ | 阻塞/异常 |
| ⏭️ | 跳过（需注明原因） |

## 规则与约束

1. **原则先行**：每次启动必须先阅读 `CLAUDE.md`，所有行为受其约束（先想后做、简单优先、最小变更、目标驱动）。
2. **需求为准**：任何实现以 `PRD.md` 为准，如有歧义与用户确认，不做主观臆断。
3. **需求变更同步**：沟通中涉及需求新增或变化时，必须同步更新 `PRD.md`，并确认设计稿是否需要更新（默认需要），与用户确认后才继续。
4. **P0 优先**：除非用户明确指定，否则始终优先推进 P0 功能。
5. **进度透明**：每次对话必须更新 `progress.md`，确保进度可追溯。
6. **设计先行**：开发前确认设计稿状态，未完成设计的功能先推动设计完成。
7. **设计稿版本管理**：修改设计稿时**禁止覆盖**现有文件，每次新建带版本号的文件：
   - 查看当前最新版本号（从 `design_preview/STATUS.md` 或目录中已有文件确认）。
   - 新建 `design_preview/index_v{新版本号}.html`（如当前最新是 v5，则新建 `index_v6.html`）。
   - 在新文件中进行所有修改，**不改动**旧版本文件。
   - 修改完成后，将 `design_preview/index.html` 更新为最新版内容（方便直接预览）。
   - 确保 `design_preview/` 目录下始终保留所有历史版本文件，便于回溯和对比。
8. **风险上报**：发现阻塞或风险时，立即记录并告知用户，不隐藏问题。
9. **验证门控**：任何阶段任务完成后必须验证通过才能更新进度为已完成，验证未通过则分析原因并继续修复，不跳过、不降标。
