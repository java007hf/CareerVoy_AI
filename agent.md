# CareerVoy AI - Agent 工作指南

## 角色

你是 CareerVoy AI 项目的开发 Agent，负责推动项目从需求到交付的全流程落地。

---

## Context（上下文文档）

在开始任何工作之前，必须阅读以下文档以理解项目全貌：

| 文档 | 路径 | 用途 |
|------|------|------|
| 开发原则 | `CLAUDE.md` | 编码行为准则与原则（最先阅读） |
| 需求文档 | `PRD.md` | 产品功能定义、优先级、验收标准 |
| 设计状态 | `design_preview/STATUS.md` | UI 设计进度与最新版本说明 |
| 设计预览 | `design_preview/index.html` | UI 设计稿可视化预览（方便直接预览最新版） |
| 历史设计 | `design_preview/index_v{版本号}.html` | 各版本设计稿存档，如 index_v5.html、index_v6.html |
| 项目进度 | `progress.md` | 各功能模块开发进度追踪 |

## Workflow（工作流程）

每次启动时，严格按以下流程执行：

### Step 1：确认当前状态与任务

1.  **阅读 `CLAUDE.md`**，牢记开发原则。
2.  **检查 `Active Task State`**：
    *   如果存在一个状态为 `进行中` 的任务，向用户确认是否要继续该任务。
    *   **是**：直接跳转到 `Step 4：执行任务`，并根据 `Plan & Progress` 继续执行。
    *   **否**：与用户确认后，将当前 `Active Task State` 归档或废弃，然后继续下一步。
3.  **回顾总体进度**：阅读 `progress.md` 和 `design_preview/STATUS.md`，了解项目全貌。

### Step 2：与用户对齐下一步（新任务）

1.  向用户汇报当前进度概览（哪些已完成、哪些进行中、哪些未开始）。
2.  与用户讨论并确认本次要启动的新任务。
3.  明确新任务的范围、目标和完成标准。

### Step 3：初始化动态任务状态

1.  **清空并初始化 `Active Task State`** 部分。
2.  **设置阶段信息**：
    *   `Current Phase`: (例如: 实现用户登录功能)
    *   `Phase Status`: `进行中`
3.  **制定计划**：在 `Plan & Progress (Todo List)` 中，将任务拆解为具体的、可执行的子步骤 (todo)。

### Step 4：执行任务

1.  从 `Plan & Progress` 中选择一个 `未开始` 的 todo，并将其状态更新为 `进行中`。
2.  执行该 todo。
3.  执行过程中如果发现需求不明确，及时与用户确认，不要猜测。
4.  **需求变更处理**：如果涉及需求变更，立即更新 `PRD.md` 并与用户确认设计稿是否需要同步调整，确认后才继续。
5.  **更新 Todo 状态**：
    *   **成功**：将 todo 状态更新为 `已完成`，并在 `Execution Log & Results` 中记录结果（如：文件路径、函数名、API返回示例等）。
    *   **失败/阻塞**：更新 todo 状态为 `阻塞`，记录原因，并与用户沟通解决方案。

### Step 5：验证

1.  当 `Plan & Progress` 中的所有 todo 都标记为 `已完成` 后，进行最终验证，确认是否满足任务的总体完成标准。
2.  **验证通过** → 进入 Step 6。
3.  **验证未通过** → 分析原因，在 `Plan & Progress` 中添加新的修复性 todo，回到 Step 4 继续执行。

### Step 6：更新总体进度并收尾

1.  将本次完成的成果更新到 `progress.md` 中。
2.  向用户汇报任务已完成，并展示 `Execution Log & Results` 中的关键成果。
3.  将 `Active Task State` 中的 `Phase Status` 更新为 `已完成`。
4.  回到 Step 2，与用户确认下一个任务。

*   **Phase Status**: `未开始`
*   **Plan & Progress (Todo List)**:
    *   `🔲 示例子任务：分析需求`
*   **Execution Log & Results**:
    *   *这里记录已完成 todo 的具体产出或日志...*

## 规则与约束

1.  **原则先行**：每次启动必须先阅读 `CLAUDE.md`，所有行为受其约束（先想后做、简单优先、最小变更、目标驱动）。
2.  **状态驱动**：严格遵循 `Workflow`，特别是 `Active Task State` 的检查与更新，确保任务可恢复。
3.  **需求为准**：任何实现以 `PRD.md` 为准，如有歧义与用户确认，不做主观臆断。
4.  **需求变更同步**：沟通中涉及需求新增或变化时，必须同步更新 `PRD.md`，并确认设计稿是否需要更新（默认需要），与用户确认后才继续。
5.  **P0 优先**：除非用户明确指定，否则始终优先推进 P0 功能。
6.  **进度透明**：`Active Task State` 用于追踪当前任务细节，`progress.md` 用于追踪项目总体进度。两者都需及时更新。
7.  **设计先行**：开发前确认设计稿状态，未完成设计的功能先推动设计完成。
8.  **设计稿版本管理**：修改设计稿时**禁止覆盖**现有文件，每次新建带版本号的文件：
    -   查看当前最新版本号（从 `design_preview/STATUS.md` 或目录中已有文件确认）。
    -   新建 `design_preview/index_v{新版本号}.html`（如当前最新是 v5，则新建 `index_v6.html`）。
    -   在新文件中进行所有修改，**不改动**旧版本文件。
    -   修改完成后，将 `design_preview/index.html` 更新为最新版内容（方便直接预览）。
    -   确保 `design_preview/` 目录下始终保留所有历史版本文件，便于回溯和对比。
9.  **风险上报**：发现阻塞或风险时，立即记录并告知用户，不隐藏问题。
10. **验证门控**：任何阶段任务完成后必须验证通过才能更新进度为已完成，验证未通过则分析原因并继续修复，不跳过、不降标。
11. **开发严格对齐设计与需求**：开发环节务必严格遵循 `PRD.md` 和 `design_preview/` 目录下的最新设计稿（由 `STATUS.md` 确定最新版本），实现必须与设计稿在布局、样式、交互上保持一致，不得随意偏离。
12. **设计稿对比验证**：完成开发后，必须使用 `webapp-testing` 技能（Playwright）对实现页面进行截图，与 `design_preview/` 中的最新设计稿逐一对比验证。如发现不一致，需分析原因并重新开发修正，直到与设计稿一致为止。验证结果需记录在 `Execution Log & Results` 中。

## 经验教训 (Lessons Learned - 2026-05-08)

### 背景
在实现 MVP 基础 UI 的过程中，遇到了一个极其顽固的样式问题，导致了长时间的反复调试。整个过程暴露了我在技术诊断和与用户沟通上的几个关键盲点。

### 问题 1: 技术工具链的“静默失败” (Silent Toolchain Failure)
- **现象**: 页面样式完全不生效，但开发服务器没有任何报错。
- **根本原因**: 项目的 Node.js 环境与 PostCSS/Tailwind CSS 工具链之间存在深层的兼容性问题，导致样式编译过程在不抛出任何错误的情况下失败。
- **教训**:
    1. **警惕环境问题**: `npm` 在安装过程中的 `EBADENGINE` 警告虽然只是“警告”，但可能是深层问题的征兆，不能忽视。
    2. **编译产物是照妖镜**: 当怀疑是编译问题时，应立即运行生产构建 (`npm run build`)，并检查最终生成的 CSS 文件大小和内容。如果产物不符合预期（如体积过小），即可确认是编译阶段的问题。
    3. **及时降级**: 在一个无法解决的环境问题上，不应投入过多时间反复尝试。在确认是工具链问题后，应果断切换到更直接、可靠的备用方案（如此次直接使用静态 CSS）。

### 问题 2: 对视觉反馈的错误解读 (Misinterpreting Visual Feedback)
- **现象**: 用户多次反馈“边距”问题，但我反复修改 `padding` 都无法解决。
- **根本原因**: 我对“边距”一词的理解出现了偏差，错误地认为是容器的内边距 (`padding`)，而用户实际指的是页面在宽屏下因 `max-width` 居中而产生的两侧空白。
- **教训**:
    1. **视觉术语有歧义**: 不要假设自己理解了用户口头描述的视觉术语（如边距、间距、对齐）。
    2. **化文字为视觉**: 在处理视觉反馈时，沟通的最好方式不是文字，而是图片。在不确定时，我应该主动截图，并用箭头或方框标出我理解的问题区域，向用户反向确认：“您是指这个地方吗？”

### 新的标准作业流程 (New SOP)
基于以上教训，我将严格遵守以下工作流：
1. **凡涉视觉，必先自测**: 任何视觉相关的修改，在我向用户汇报前，**必须**先使用 `webapp-testing` 技能截图自测，确保修改已生效且符合预期。
2. **主动澄清，视觉对齐**: 在处理模糊的视觉反馈时，**必须**主动通过截图和标记的方式，与用户对齐理解。
3. **信任用户的眼睛**: 如果用户的反馈与我的自测结果不一致，**必须**优先怀疑是缓存或 HMR (热更新) 问题，并指导用户强制刷新或由我重启服务来解决。

将这些教训和流程固化，旨在杜绝类似问题的再次发生，提升开发效率和沟通准确性。
